% ========== 1. INTRODUCTION ==========
\section{Introduction}

\subsection{Purpose}
This document defines the test cases required to verify that the \textbf{mail deletion} feature of a web-based mail provider (modeled after Google Mail) functions correctly and meets expected behavior. The test cases are designed to ensure comprehensive coverage of positive, negative, and edge-case scenarios.

\subsection{Scope}
The scope of testing covers:
\begin{itemize}[noitemsep]
    \item Deleting a single email from the Inbox
    \item Deleting multiple emails (bulk deletion)
    \item Deleting emails from different views (Inbox, Sent, Drafts, Spam, custom labels)
    \item Verifying deleted emails appear in Trash
    \item Permanent deletion from Trash
    \item Undoing a deletion
    \item Auto-purge of Trash after 30 days
    \item Keyboard shortcut-based deletion
    \item Deletion behavior for starred/important emails
    \item Deletion behavior for emails with attachments
    \item UI validation (visual feedback, confirmation dialogs)
\end{itemize}

\subsection{Test Case Format}
Each test case follows a structured format with the fields described in Table~\ref{tab:format}.

\section{Risks and Assumptions}

\begin{itemize}[noitemsep]
    \item \textbf{Risk:} Gmail UI may change frequently; locators used in automated tests may break.
    \item \textbf{Risk:} Google may enforce CAPTCHA or additional verification (e.g., OTP), blocking automated login.
    \item \textbf{Assumption:} The test Gmail account has been pre-configured and contains the necessary test data.
    \item \textbf{Assumption:} Tests are executed on the latest stable version of Google Chrome.
    \item \textbf{Assumption:} Network connectivity is stable unless explicitly testing network failure scenarios.
    \item \textbf{Mitigation:} For Google verification challenges, use environment-configurable credentials and consider using Google OAuth or App Passwords for automation.
\end{itemize}
